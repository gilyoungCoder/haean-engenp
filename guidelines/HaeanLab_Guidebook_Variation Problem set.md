## PART 1. 공통 제작 원칙

### 1-1. 지문 구성 원칙

1. **유형당 4문제 구성**: 원문 유지 2문 + 패러프레이징 2문
2. **원문 유지 문제**: 지문의 어휘·구문·문장 순서를 그대로 사용. 출제 포인트(밑줄/빈칸)만 설계
3. **패러프레이징 문제**: 동일 의미를 수능 수준의 어휘와 구문으로 재서술

> ⚠️ **패러프레이징 3대 절대 금지**
> 1. 지문에 없는 새로운 정보(인물·사건·수치·배경) 추가 금지
> 2. 지문 문장의 단어·내용 누락 금지 (의미 손실 불허)
> 3. 핵심 메시지·결말·교훈을 바꾸는 의역 금지

### 1-2. 난이도 기준

- 수능/모의고사 **3~4등급** 목표
- 한 유형 내 정답 포인트 **중복 금지**: 어법 4문 → 수일치·준동사·관계사·분사 등 서로 다른 범주
- 오답은 **'본문 어휘 활용 + 실제처럼 보이는 함정'** 구조 — 단순 반의어 지양

### 1-3. 출력 구조 (핵심 규칙)

**각 유형 출력 내부 구조 (수동/API 공통):**

```
[문제 파트]
  └ 문항 1번 (지문 + 선지 + 서술형 답란)
  └ 문항 2번
  └ 문항 3번
  └ 문항 4번

[해설 파트]  ← 반드시 문제가 모두 끝난 뒤에 배치
  └ 정답표 (1~4번)
  └ 1번 해설
  └ 2번 해설
  └ 3번 해설
  └ 4번 해설
```

> ⚠️ **문항-해설 교차 배치 절대 금지**  
> `문항1 → 해설1 → 문항2 → 해설2` 순서는 오답입니다.  
> 반드시 **문제 전부 → 해설 전부** 순서로 출력하세요.  
> API 자동화 시에는 JSON 구조 자체가 이 분리를 보장합니다. (PART 8 참고)

### 1-4. 밑줄 표기 규칙 (API 자동화 핵심)

수동 제작과 API 자동화 모두 **`[[텍스트]]` 태그**로 밑줄 위치를 표시합니다.

```
수동 제작 시 → Claude가 지문에 [[텍스트]] 태그 삽입
API 자동화 시 → JSON passage 필드에 [[텍스트]] 태그 포함
              → 백엔드에서 태그 파싱 후 python-docx underline 속성 적용
```

**예시:**
```
원문: "The benefits is not limited to physical strength"
태그: "The benefits [[is]] not limited to physical strength"
→ docx: "is" 부분에 underline 적용
```

## PART 2. 지문 분석 → 문항 설계 절차

```
STEP 1. 지문 분석
  ├ 전체 서사/논지의 흐름 파악 (도입 → 전개 → 결말/교훈)
  ├ 핵심 어휘·구문 목록 추출 (출제 포인트 후보군)
  └ 단락별 주요 사실·수치·고유명사 정리 (내용일치 대비)
          ↓
STEP 2. 문항 설계
  ├ 유형별 정답 포인트 4종 선정 (중복 없이 다양하게)
  ├ 원문 2문: 밑줄/빈칸 위치 지정, 오답 3~4개 설계
  └ 패러프레이징 2문: 재서술 후 3대 금지 원칙 자가 검토
          ↓
STEP 3. 검토 & 출력
  ├ PART 4 체크리스트 전 항목 점검
  ├ 정답 1개 / 오답 근거 명확성 최종 확인
  ├ [수동] docx 생성 (문제 전부 → 해설 전부 순서 확인 필수)
  └ [API] JSON 유효성 검증 후 백엔드에서 docx 변환
```

## PART 3. 유형별 설계 공식

### 3-1. 어법 (Grammar)

**발문**: `"다음 글의 밑줄 친 부분 중 어법 상 옳은/틀린 것은?"`

**설계 절차**

1. 어법 오류를 심을 위치 5곳 선정 (문장 내 핵심 구조 자리)
2. 정답 포인트 4종 이상 분산 배치:
   - **수일치**: 수식어구가 끼어든 긴 주어 → 동사 수 혼동 유도
   - **준동사**: to V ↔ V-ing / 전치사 뒤 동명사 / 사역·지각동사 OC
   - **관계사**: who·which·whose·where·what / 계속적 용법(that 불가)
   - **분사**: 현재분사(능동) ↔ 과거분사(수동) / 감정 유발 vs 느끼는 주체
   - **형부사**: 2형식·5형식 보어 자리(형용사) vs 동사 수식 자리(부사)
   - **병렬구조**: and/or 연결 시 동일 품사·형태 유지
3. 정답 1개: 원문 그대로 어법상 옳은 형태 유지
4. 오답 4개: 각각 명확한 교체 형태 설정

> **고난도 장치 3가지**
> 1. 주어가 관계절·분사구·전치사구로 확장된 경우 수일치 오류 유도
> 2. 계속적 용법 관계절에 `that` 배치 → `which` 교체 필요
> 3. 지각동사 `find/see` + 목적어 + OC(현재분사/과거분사) 혼동

**API JSON 핵심 필드**: `passage`(밑줄 `[[]]` 태그), `underline_labels`, `choices`, `answer`, `grammar_point`, `explanation.correct`, `explanation.wrong` → PART 8 스키마 참고

---

### 3-2. 어휘 (Vocabulary)

**발문**: `"문맥 상 낱말의 쓰임이 적절한/적절하지 않은 것은?"`

**설계 절차**

1. 지문의 흐름(긍정↔부정 / 원인↔결과)을 파악하고 키워드 5곳 선정
2. 정답 오류어 1개: 문맥과 의미가 반대이거나 어울리지 않는 단어로 교체
   - 단순 반의어 지양 → 상위 개념·학술어 수준 (예: `scattered → cherished`)
   - 교체 단어의 **품사는 반드시 원문과 동일**하게 유지
3. 오답(적절한 단어) 3~4개: 본문 흐름과 자연스럽게 어울리는 단어 유지

> **어휘 출제 4대 패턴**
> 1. 감정 반전: `disappointed → delighted` / `welcomed → alienated`
> 2. 행위 반전: `scattered → cherished` / `connect → disengage`
> 3. 정도 반전: `countless → limited` / `overwhelming → manageable`
> 4. 의미 혼동: `concealment`(은폐) vs `reminder` / `speculate`(추측) vs `reflect`

**API JSON 핵심 필드**: `error_word`, `correct_word` 필드를 별도로 반환 → docx 생성 시 오류어 강조 처리 가능

---

### 3-3. 대의파악 (Main Idea)

**4문항 구성**: 주제 / 제목 / 요약문 완성 / 필자의 주장 — 혼합 구성

| 세부 유형 | 설계 포인트 | 오답 설계 |
|-----------|-------------|-----------|
| 주제/주장 | 핵심 메시지가 선지에 추상화 | '부분 사실'(배경·소재·소품)로 혼동 |
| 제목 | 상징어 포함 필수 | 표면 사건만 묘사하거나 부분 소재만 강조 |
| 요약문 (A)(B) | (A)=핵심 행위·수식어(형용사), (B)=결과·교훈(추상 명사) | (A): 지문 소재어 / (B): 중간 과정 감정어 |
| 필자 주장 (한국어 선지) | 마지막 문장·반복 어구에서 추출 | 유사 주제 영역으로 혼동 |

**API JSON 핵심 필드**: 요약문 문항의 경우 `summary_template`, `(A)`, `(B)` 정답을 별도 필드로 분리 → 빈칸 렌더링에 활용

### 3-4. 문장 삽입 / 글의 순서 (Flow)

**문장 삽입 — 단서 3종**

- **지시어 단서**: 삽입 문장 내 `this/these/the+명사` → 앞 문장에 선행 명사 필수
- **연결어 단서**: `However·Yet`(역접) / `Therefore`(결과) / `Furthermore`(추가) → 앞뒤 논리 관계 명확화
- **시간/인과 단서**: `Before long·After+동명사·went on his way` → 삽입 전후 행위 시간 순서 일치

**글의 순서 — 연결 고리 설계**

- 각 단락의 첫 문장(도입) ↔ 끝 문장(연결)을 연결 고리로 설계
- 주어진 글 마지막 명사/사건이 (A)/(B)/(C) 중 하나의 첫 문장과 직결
- **고난도 장치**: 주어진 글이 서사 중반 시작 → (C)가 주어진 글보다 시간적으로 앞선 맥락 → 역방향 추론 필요

**API JSON 핵심 필드**: 삽입형은 `insert_sentence` + `passage_with_markers`(①~⑤ 위치 표기), 순서형은 `given_passage` + `blocks.A/B/C`를 분리 반환

---

### 3-5. 서술형 (Written Response)

**① 어법 오류 3개 고치기**

- 밑줄 3개: 모두 어법 오류, **서로 다른 문법 범주**에서 선정
- 답란: `번호 | 밑줄 표현 | 고친 표현`의 3열 표 제공
- 해설: 오류 형태 → 올바른 형태 + 문법 규칙 근거 명시

**② 요약문 빈칸 (A)(B)**

- 조건 문구: **"본문에서 찾아 필요시 변형하여 쓰시오"**
- `(A)`: 본문 핵심 동사·형용사 → 시제·형태 변형 (예: `recognize → recognized`)
- `(B)`: 본문 핵심 명사·동사 → 원형 또는 과거형 (예: `scatter → scattered`)
- 답란: (A) / (B) 2칸 표 제공

**API JSON 핵심 필드**: `errors[].wrong`, `errors[].correct`, `errors[].reason` 배열로 분리 → 3열 표 자동 생성. 요약빈칸은 `answers.A.word`, `answers.A.base_form`, `answers.A.transform` 분리

### 3-6. 내용 일치 / 불일치 (Inference)

**발문**: `"본문의 내용과 일치하는/일치하지 않는 것을 고르시오"` (영어 선지 구성)

**설계 절차**

1. 지문에서 검증 가능한 구체적 사실 5개 선정: 인물 행동·장소·수치·순서·특징
2. 정답 불일치 선지 1개: 지문의 사실을 아래 방법으로 변형
   - **주어 교체**: A의 행동을 B의 것으로 교체
   - **서술어 반전**: 원문의 동사·형용사를 반대 의미로 교체
   - **조건 변형**: '~할 수 있다'를 '~할 수 없다'로 / '~만'을 '~도'로
   - **시점 혼동**: 과거 사실을 현재 계획처럼 서술
3. 오답(일치 선지) 4개: 본문 문장을 영어로 paraphrase하여 구성
4. 해설: 정답 선지의 불일치 근거 → 본문 원문 인용하여 명시

> **영어 선지 고난도 설계 팁**
> - 일치 선지를 원문과 다른 어휘로 paraphrase → 낚시 효과
> - 불일치 선지는 원문의 일부 표현을 차용 → 표면적 유사성 증가
> - 수치·시간·조건 변형이 가장 정확한 불일치 포인트

**API JSON 핵심 필드**: `false_type`(주어교체/서술어반전 등), `evidence`(본문 원문 근거) 분리 → 해설 자동 생성에 활용

- **Q1 주제**: 오답 = 여정 묘사(①) / 부모 교훈(②) / 자연 묘사(④) / 우정(⑤)
- **Q3 요약문**: (A) `glittering` 소재어 혼동 / (B) `disappointment` 중간 감정 혼동

### 순서/삽입 — 핵심 단서

- **Q1 삽입 ④**: `the remaining pieces`(식사 선행 필수) + `went on his way`(이동 재개)
- **Q4 순서 ② (C→A→B)**: 주어진 글 이전 맥락을 (C)로 역방향 추론 — 최고난도

### 서술형 — 문법 범주 배분

- **Q1**: 전치사+동명사 / 불필요한 비교급 / 소유격 관계대명사
- **Q4 요약문**: `recognize→recognized` / `scatter→scattered`

def validate(result: dict, q_type: str):
    # 1. 필수 최상위 필드
    assert "type"      in result
    assert "questions" in result

    # 2. 문항 수
    assert len(result["questions"]) == 4, f"문항 수 오류: {len(result['questions'])}개"

    for q in result["questions"]:
        # 3. 공통 필수 필드
        assert "passage"     in q
        assert "answer"      in q
        assert "explanation" in q

        # 4. 유형별 추가 검증
        if q_type in ["어법", "어휘", "대의파악", "순서삽입", "내용일치"]:
            assert len(q.get("choices", [])) == 5, "선지 5개 필수"

        if q_type == "어법":
            assert "grammar_point" in q
            # [[]] 태그 개수 = 선지 수 검증
            import re
            tags = re.findall(r"\[\[.+?\]\]", q["passage"])
